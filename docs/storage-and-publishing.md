# 回响册储存与发布策略

## 结论

GitHub 适合发布开源代码和演示应用，不适合保存用户的票根、海报、座位图和现场照片。个人数据采用“元数据在数据库、图片在对象存储、本地保留完整备份”的结构。

## 当前默认方案

- 应用代码：GitHub + GitHub Pages
- 个人元数据：Supabase Database
- 个人图片：Supabase Storage，当前项目默认桶为 `private-data`
- 本地离线：IndexedDB
- 灾备：应用内完整 JSON 导出

## 为什么不把图片放进 GitHub

GitHub Pages 是静态站点托管，适合 HTML/CSS/JS；GitHub 仓库对大文件和仓库体积也有明确限制。个人图片进入 git 后还会留下历史记录，不利于隐私和长期维护。

## Supabase 免费空间不够怎么办

几十 MB/人的图片非常正常，尤其是保存电子票根、座位图和现场照片后会快速增长。解决顺序如下：

1. **前端压缩**
   - 上传前限制长边，例如 1800px。
   - 海报和座位图保留较高清，现场精选照片压得更小。
   - 当前应用已经在 `src/media.ts` 做上传前压缩。

2. **分级保存**
   - 应用里只保存“可浏览版”：海报、票根、座位图、精选照片。
   - 原图保存在手机相册、网盘或硬盘。
   - 每场建议：1 张海报/票根 + 1 张座位图 + 3-6 张现场精选。

3. **对象存储可替换**
   - 默认 Supabase Storage。
   - 预留 Cloudflare R2、腾讯云 COS、阿里云 OSS、S3 兼容存储。
   - 数据库只保存媒体引用，因此以后迁移对象存储不会破坏记录结构。

4. **私有同步与公开展示分离**
   - 私人档案用私有桶和签名 URL。
   - 如果以后做公开展示页，另建公开缩略图桶，只放低清封面。

## 推荐的长期结构

```text
GitHub Pages
  只放应用代码、图标、示例数据

Supabase Database
  echo_records: 演出记录 JSON
  echo_media_assets: 图片索引、宽高、大小、来源

Object Storage
  userId/recordId/mediaId.jpg
  保存票根、海报、座位图、现场照片

Local IndexedDB
  离线缓存和未同步队列
```

## 小程序路线

微信小程序很适合“打开快、手机端沉浸、随时翻票根”。但它不是把当前网页直接发布一下那么简单。

### 最省事：WebView 小程序

- 继续维护 GitHub Pages / 独立域名的 Web App。
- 小程序里用 `web-view` 打开网页。
- 优点：改动最小，复用当前 UI 和 Supabase。
- 缺点：需要合规域名、HTTPS、微信业务域名配置；体验仍然是网页壳。

### 更像原生：Taro / uni-app 重写移动端

- 复用现有 TypeScript 数据模型和 Supabase/对象存储思路。
- UI 用小程序组件重写。
- 优点：体验最好，图片选择、分享、扫码更自然。
- 缺点：工作量明显增加，桌面 Web 和小程序需要维护两套展示层。

### 推荐路线

先把 Web 版发布稳定，再做小程序：

1. 发布 GitHub Pages + Supabase 私人同步。
2. 增加对象存储适配器，控制图片体积。
3. 若只给自己用，先做 WebView 小程序壳。
4. 如果后续要认真发布给别人用，再用 Taro/uni-app 做原生小程序端。
