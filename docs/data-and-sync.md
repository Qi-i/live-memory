# 数据与同步

## 三层数据

| 数据层 | 保存内容 | 图片 | 跨设备 |
| --- | --- | --- | --- |
| 当前设备 | 完整记录与本地图片 | 完整 | 否 |
| Live Memory 账号 | 账号资料与演出文字备份 | 不保存 | 是 |
| 个人 Supabase | 完整记录 | 可选 | 是 |

所有新建和编辑操作先写入当前设备的 IndexedDB。云端操作失败时，本地档案仍然保留。

## Live Memory 账号

账号项目由站点维护者配置，负责：

- 用户名、昵称、压缩头像和找回邮箱。
- 密码登录、GitHub 登录和密码找回。
- `echo_text_backups` 演出文字备份。
- 用户自己的个人 Supabase 公开连接配置。

找回邮箱只用于 Live Memory 密码找回，不会写入用户个人 Supabase 设置。

文字备份会移除整个 `media` 数组，因此不会上传海报、票根、座位图和现场照片。恢复文字备份时，同 ID 记录的本地图片会继续保留。

## 个人 Supabase

个人 Supabase 是用户自己创建和管理的完整档案空间：

| 表或空间 | 内容 |
| --- | --- |
| `echo_passkey_records` | 演出记录 JSON、更新时间和删除状态 |
| `echo_passkey_media_assets` | 图片类型、尺寸和存储路径 |
| `echo-media` | 私有图片文件 |

公开站点登录账号后，个人云端通过账号会话和 Supabase 项目地址生成连接钥匙。没有账号服务的自部署版本，会通过“用户名 + 档案密码 + Supabase 项目地址”生成连接钥匙。关闭“同步图片”后，完整同步只写演出文字；恢复时与当前设备的图片合并。开启后，本地 `data:` 图片会先上传到私有图片空间，再把存储路径写入记录。

## 自动备份

自动备份只处理文字资料。用户可以选择每小时、每 6 小时、每天或每周执行。档案发生变化且达到设定间隔后，页面会延迟 30 秒写入账号备份。

自动备份需要有效的 Live Memory 登录会话。登录过期时不会删除或覆盖当前设备的数据，重新登录后可手动执行“立即备份文字”。

## 回收站

第一次删除会写入 `deletedAt` 并把记录移入回收站。该状态会进入文字备份和完整同步，防止另一台设备重新带回已删除记录。

- 恢复：清除 `deletedAt`，记录回到档案。
- 永久删除：再次确认后删除本地记录；已连接个人 Supabase 时，同时删除云端记录和对应图片目录。

## 冲突规则

文字恢复按 `updatedAt` 比较同 ID 记录：

- 云端较新：采用云端文字字段，保留当前设备图片。
- 本地较新：保留本地记录。
- 仅云端存在：创建一条不含图片的本地记录。
- 仅本地存在：保留本地记录，等待下次备份。

完整图片同步仍采用用户主动上传和恢复。跨设备同时编辑同一条记录前，先执行一次恢复，再编辑并上传。

## 文件备份

- 完整 JSON：包含记录和本地图片，适合迁移与灾备。
- 文字 JSON：不含图片，文件小，适合频繁保存。
- CSV：适合表格查看和统计，不含图片。

完整 JSON 可能含票根二维码、订单信息和现场照片，应保存在私人设备或可信存储中。
